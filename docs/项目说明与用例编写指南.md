# 接口自动化项目：说明与用例编写指南

本文档面向在本仓库中**启动环境**、**编写用例**以及**迭代框架**的同学，说明当前工程的运行机制、推荐范式与扩展步骤。

---

## 1. 项目是什么

本仓库是一个 **Python 接口自动化测试工程**，技术栈为：

| 组件 | 用途 |
|------|------|
| pytest | 测试框架与收集执行 |
| requests | HTTP 调用 |
| PyYAML | 流程与数据定义 |
| allure-pytest | 报告附件（请求/响应） |
| SQLite | 本地库表初始化与结果校验 |
| Faker | 测试数据工厂 |
| jsonpath-ng | 响应提取（JSONPath） |
| jsonschema | 契约校验 |

业务上假定被测对象是「电商相关接口」。当前仓库内置 **`mock_server.py`（Flask）** 模拟本地电商 API，便于**不依赖外部服务**即可跑通用例。若你对接真实后端，只需保证 `config/config.yaml` 中的地址与鉴权方式一致，并可关闭或绕过 Mock。

### 1.1 企业级特性

- ✅ **多环境配置**：dotenv + env 分层（dev/test/prod），环境变量优先级最高
- ✅ **统一日志**：文件轮转 + 级别控制 + token 掩码
- ✅ **请求增强**：超时/重试（指数退避）/异常统一/日志掩码
- ✅ **Runner 增强**：setup/teardown、条件跳过（when）、循环（loop）、软断言（soft_assert）、JSONSchema 校验
- ✅ **测试分层**：smoke / regression / concurrency markers，CI 分阶段执行
- ✅ **数据工厂**：Faker 生成唯一测试数据，避免冲突
- ✅ **JSONPath 提取**：支持 `$..id` 等复杂路径
- ✅ **契约校验**：关键接口响应自动校验 JSONSchema
- ✅ **CI/CD**：GitHub Actions（lint + smoke/regression 分阶段）
- ✅ **Mock 目录化**：按资源拆分路由（mocks/routes/）

## 3. 启动「电商」服务与测试（两种方式）

### 3.1 推荐：直接跑 pytest（一体化）

`conftest.py` 会在**会话开始时**自动：

1. 删除并重建 `test.db`（路径见配置），执行 `init_db()` 建表；
2. 向 `users` 表写入配置中的默认账号（`admin` / `admin123`）；
3. 在后台线程启动 Mock 服务（默认 `http://127.0.0.1:5000`）；
4. 注册默认登录函数并调用 `TokenManager.get_token()`，供多数带鉴权的接口使用。

因此通常**不需要**单独先敲命令启动电商项目，只需：

```bash
cd /path/to/APIAuto
python3 -m pip install -r requirements.txt
python3 -m pytest
```

`pytest.ini` 已包含 `--alluredir=report`，因此**每次**执行上面命令都会在项目根目录生成 **`report/`**（Allure 原始结果）。如何在浏览器里打开 HTML 报告，见 **第 4 节**。

### 3.2 可选：仅调试 Mock 接口时手动启动

若你想用 Postman/curl 单独调接口：

```bash
cd /path/to/APIAuto
python3 -m pip install -r requirements.txt
# 先保证 test.db 已存在且表已建好：可先跑一次 pytest，或在本机执行 init_db 逻辑
python3 mock_server.py
```

默认监听 `127.0.0.1:5000`。注意：**手动启动前**若与 pytest 同时占用 5000 端口会冲突，测试时请只保留一种启动方式。

### 3.3 端口被占用（常见：macOS）

若 `5000` 被占用，可修改：

- `config/config.yaml` 中 `base.url`（例如改为 `http://127.0.0.1:5001`）；
- `conftest.py` 中 `make_server("127.0.0.1", 5000, ...)` 的端口，与配置保持一致。

也可用环境变量覆盖基地址（见第 5 节）。

## 4. Allure 报告（先看这段避免混淆）

Allure 在本项目里分两步理解，不要和「装 Python 依赖」混为一谈：

| 步骤 | 谁来做 | 产出 |
|------|--------|------|
| ① 跑测试 | `pip` 安装的 **`allure-pytest`** + pytest | 在项目根目录写入 **`report/`** 目录（JSON 等原始结果，不是网页） |
| ② 看 HTML | 单独安装的 **Allure 命令行（CLI）** | 本地起一个服务或生成静态站点，浏览器里看报告 |

### 4.1 第一步：pytest（每次测试都要先做）

在项目根目录执行：

```bash
python3 -m pytest
```

由于 `pytest.ini` 里已配置 `addopts = ... --alluredir=report`，无需再手写 `--alluredir`（除非你刻意想换目录名）。跑完后应存在 **`report/`**，否则检查用例是否执行成功、当前工作目录是否为项目根。

若你希望显式指定目录（与默认等价）：

```bash
python3 -m pytest --alluredir=report
```

### 4.2 第二步：allure serve（生成并浏览 HTML）

**`allure` 不是 pip 包**，需要本机安装 Allure CLI，例如 macOS：

```bash
brew install allure
```

安装成功后，**在第一步生成 `report/` 之后**，在同一项目根目录执行：

```bash
allure serve report
```

会在本机启动临时 Web 服务并自动打开浏览器；关闭终端或中断命令即停止服务。

若只需要生成静态页面到某目录（便于拷贝或 CI 归档）：

```bash
allure generate report -o allure-report --clean
```

然后用浏览器打开 **`allure-report/index.html`**。

### 4.3 小结

- **只有 pytest**：可以，会得到 **`report/`** 原始数据，适合 CI 归档或由 Allure Server 消费。
- **没有安装 Allure CLI**：**无法**通过 `allure serve` 一键看 HTML；可先保留 `report/` 目录，装好 CLI 后再执行第二步。
- **`common/request_util.py`** 里对请求/响应的 Allure 附件，会进入第一步产生的原始结果中。

## 5. 配置说明

### 5.1 `config/config.yaml`

- **`base.url`**：所有 API 的基地址，应与 Mock 或真实服务一致。
- **`database.path`**：SQLite 文件相对项目根目录的路径（如 `test.db`）。
- **`user.username` / `user.password`**：默认登录账号，用于 `common/auth.py` 中的 `default_login()`，并在会话初始化时种子写入 `users` 表。
- **`timeout`**：请求超时（秒），默认 30。
- **`retries`**：请求失败重试次数，默认 0。
- **`log.level`**：日志级别（DEBUG/INFO/WARNING/ERROR）。

### 5.2 环境变量（可选）

- **`API_ENV`**：环境标识（dev/test/prod），默认 `dev`。
- **`API_BASE_URL`**：覆盖 `config.yaml` 里的 `base.url`。
- **`API_USERNAME` / `API_PASSWORD`**：覆盖默认登录账号。
- **`API_REQUEST_TIMEOUT`**：覆盖超时配置。
- **`API_REQUEST_RETRIES`**：覆盖重试配置。
- **`LOG_LEVEL`**：覆盖日志级别。

### 5.3 环境覆盖配置

在 `config/env/` 目录下可放置 `dev.yaml` / `test.yaml` / `prod.yaml`，会自动与 `config.yaml` 合并（环境配置优先级更高）。

优先级：环境变量 > env/<API_ENV>.yaml > config.yaml

## 6. 运行时你在「跑什么」

一条简化链路如下：

```text
pytest 启动
  → pytest_sessionstart：删库、建表、插入默认用户
  → session fixture：启动 Mock → TokenManager 注册登录 → 预取 token
  → testcase：YAML 流程 / 纯 Python 用例
       → API 层 → RequestUtil（自动 Bearer token）→ HTTP
       → extract / assert / db 查询
```

- **默认鉴权**：`RequestUtil` 在未设置 `no_token=True` 时，从 `TokenManager` 取 token，请求头为 **`Authorization: Bearer <token>`**。若真实后端格式不同，应在 `request_util.py` 统一修改。
- **YAML 流程里登录换账号**：`extract` 出 `token` 后，`runner` 会调用 `TokenManager.set_token()`，后续步骤使用该 token。

## 7. 编写用例：两种范式

### 7.1 范式 A：YAML 数据驱动（回归业务链）

适用：**步骤固定、希望数据与代码分离**的场景（注册→登录→下单→支付等）。

1. 在 `data/flow.yaml` 的 `cases` 下新增一条 `name` + `steps`。
2. `testcase/test_flow.py` 已参数化加载该文件，保存后执行 pytest 即可跑到新用例。

**单步字段约定：**

| 字段 | 含义 |
|------|------|
| `api` | 注册在 `common/api_factory.py` 中的名称（见下节） |
| `data` | 传给对应 API 方法的参数字典（支持 `${变量}` / `${fake.xxx}`） |
| `no_token` | `true` 时跳过自动带 token（注册/登录等） |
| `extract` | 从**响应 JSON** 按路径取值写入上下文；键为上下文名，值为响应中的路径（点号分层如 `id`，或 JSONPath 如 `$..id`） |
| `assert` | 对**响应 JSON** 做子集断言（期望键值须与实际一致） |
| `schema` | （可选）JSONSchema 名称，对应 `config/schemas/<name>.json` |
| `when` | （可选）条件表达式，如 `${env} == 'prod'`，为 false 时跳过该步骤 |
| `loop` | （可选）循环次数（int）或列表（list），展开为多次请求 |
| `soft_assert` | （可选）`true` 时断言失败不中断，汇总报告 |

**Case 级字段：**

| 字段 | 含义 |
|------|------|
| `setup` | 步骤列表，在主 steps 前执行 |
| `teardown` | 步骤列表，在 case 结束时执行（即使失败也执行） |

**变量替换**：在 `data` / `assert` 中可使用 `${product_id}` 等形式，由 `replace_util` 从 `Context` 取值后再发起请求。
也支持 `${fake.username}` 等 Faker 占位，会自动生成唯一测试数据。

当前已注册的 `api` 名称：

- `user.register`、`user.login`
- `product.add_product`
- `order.create_order`
- `pay.pay_order`

扩展示例见第 8 节。

### 7.2 范式 B：Python 代码编排（并发、复杂逻辑）

适用：**多线程/进程、复杂分支、与 DB 组合断言**等 YAML 不好表达的场景。

参考 `testcase/test_oversell.py`：

- 直接使用 `ProductApi`、`OrderApi` 等；
- 会话级已有默认 token，一般无需手写登录（除非要换用户）；
- 失败请求会 `raise_for_status()`，可按 `requests.HTTPError` 处理预期失败；
- 使用 `common/db_assert.py`、`common/db_util.py` 查 SQLite 做后置校验。

## 8. 扩展一个新接口（端到端）

以下顺序可尽量保持职责清晰、避免「只在 YAML 里写了 api 却未注册」的问题。

1. **`api/`**  
   在对应资源类中新增方法：只负责路径、方法、body/query，**不写测试断言逻辑**。基类通过 `ConfigUtil.get_base_url()` 拼 URL，避免硬编码。
2. **`common/api_factory.py`**  
   在 `_REGISTRY` 中增加 `"模块语义.方法名": lambda **kw: XxxApi().method(**kw)`，名称需与 YAML 里 `api:` 一致。
3. **`data/flow.yaml`**（若走流程）  
   增加步骤，必要时配置 `extract` / `assert`。
4. **`mock_server.py`**（若仍使用内置 Mock）  
   增加路由与数据库操作，使自动化可验收。
5. **回归**  
   `python3 -m pytest`。

## 9. 数据库与断言

- **表结构**：见 `common/db_init.py`（`users`、`products`、`orders`）。
- **何时用 DB**：接口返回不足以证明状态时（库存、订单状态等），用 `db_util.query_one` / `db_assert.get_product_stock` 等查询 `test.db`。
- **注意**：每次 pytest 会话开始会**删除并重建** `test.db`，不要在同一次会话外依赖上一次运行的数据。

## 11. 常见问题

| 现象 | 排查 |
|------|------|
| `Connection refused` | Mock 未启动或 `base.url` 错误；端口是否与 `conftest` 一致 |
| `401/403` | 是否忘记 `no_token`；Bearer 格式是否与后端一致 |
| `Address already in use` | 5000 被占用，改端口或关闭占用进程 |
| YAML 步骤找不到 api | 检查 `api_factory._REGISTRY` 是否已注册名称 |
| `${var}` 报错 | 上一步是否 `extract` 了对应变量名 |
| 只有 `report/` 文件夹，打不开 HTML | 先确认已跑过 pytest；再安装 Allure CLI 后执行 `allure serve report`（见第 4 节） |
| `allure: command not found` | 未安装 Allure 命令行，macOS 可用 `brew install allure` |
| `API_ENV=test` 未生效 | 检查 `config/env/test.yaml` 是否存在；启动日志会打印当前环境 |
